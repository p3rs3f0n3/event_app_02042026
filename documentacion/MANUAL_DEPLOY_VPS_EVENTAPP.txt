DESPLIEGUE / ACTUALIZACION DEL BACKEND
CAMINO RECOMENDADO
- Usar `scripts/safe_deploy_vps_backend.py`.
- Este script fue pensado para forzar el deploy no destructivo paso a paso.
- Hace verificaciones, backup remoto, sincronizacion de `backend/` y `deploy/`, rebuild solo de `eventapp-backend` y healthcheck.
- No ejecuta `db:init`, no hace restore, no hace truncate y no toca la base de datos.

1) Conectarse a la VPS y ubicarse en el proyecto:
   cd /root/projects/eventapp

2) Subir los archivos modificados del backend.
   Opciones:
   - git pull, solo si el repo remoto esta limpio y alineado
   - SFTP/SCP, si se esta sincronizando desde local
   - script Python + Paramiko, si hace falta automatizar la carga

2.1) Antes de sobrescribir codigo remoto, verificar si el repo de la VPS tiene cambios locales:
   git status --short
   - Si hay cambios locales, NO hacer `git pull` a ciegas.
   - Hacer backup del codigo remoto y luego sincronizar solo los archivos necesarios.

2.2) Backup minimo recomendado antes de actualizar backend/deploy:
   mkdir -p /root/backups
   tar -czf /root/backups/eventapp_code_backup_<timestamp>.tar.gz -C /root/projects eventapp/backend eventapp/deploy

3) Reconstruir solo el backend:
   docker compose -f deploy/docker-compose.vps.yml --env-file deploy/eventapp.vps.env up -d --build eventapp-backend

4) Confirmar que el stack quede arriba:
   docker compose -f deploy/docker-compose.vps.yml --env-file deploy/eventapp.vps.env ps

5) Validar salud desde la VPS o desde local:
   curl https://66.94.101.47.sslip.io/api/health
   Respuesta esperada:
   {"status":"ok","repositoryDriver":"postgres"}

6) Si hace falta revisar logs del backend:
   docker logs eventapp-backend --tail 200

7) Si el problema parece de datos o migracion:
   - NO ejecutar `docker exec -it eventapp-backend npm run db:init` en una VPS con datos que se deban preservar.
   - Ese comando puede reaplicar seed y pisar informacion operativa.
   - Si hace falta una migracion, aplicar solo cambios SQL puntuales y no destructivos, con backup previo.

SCRIPTS Y COMANDOS: QUE SE PUEDE USAR Y QUE NO

SEGUROS PARA DEPLOY NORMAL
- `docker compose -f deploy/docker-compose.vps.yml --env-file deploy/eventapp.vps.env up -d --build eventapp-backend`
  - Reconstruye solo el contenedor backend.
  - No resetea la base por si mismo.
- Sincronizacion por SFTP/SCP/Paramiko de `backend/` y `deploy/`
  - Siempre con backup previo del codigo remoto.

PELIGROSOS / PROHIBIDOS EN DEPLOY NORMAL
- `docker exec eventapp-backend npm run db:init`
  - Peligro: ejecuta `backend/scripts/init-postgres.js`.
  - Ese script aplica `01_schema.sql` Y `02_seed.sql`.
  - El seed puede pisar informacion existente.
  - Desde ahora exige confirmacion explicita por variables de entorno antes de correr.
- `scripts/deploy_vps_terms_fix.py`
  - Peligro: incluye `docker exec eventapp-backend npm run db:init`.
  - NO usar este script en una VPS con datos reales.
  - Desde ahora exige flag + variable de entorno de confirmacion.
- `scripts/restore_vps_from_backup_and_migrate_terms.py`
  - Peligro MAXIMO: contiene `DROP DATABASE IF EXISTS event_app;`.
  - Solo podria usarse en un procedimiento de restauracion explicitamente aprobado y con backup verificado.
  - Desde ahora exige flag + variable de entorno de confirmacion.
- `TRUNCATE`, `DROP DATABASE`, restauraciones completas o seeds operativos
  - Prohibidos en deploy normal.

REGLA DE ORO
- Si la tarea es "actualizar backend en VPS", la accion correcta es: sincronizar codigo + reconstruir `eventapp-backend` + validar health.
- Si una accion modifica datos, recrea base, reaplica seed o restaura backup, YA NO es deploy normal: requiere aprobacion explicita, backup nuevo y validacion manual.

BARRERAS TECNICAS NUEVAS
- `npm run db:init` ahora falla por defecto salvo que se definan:

  - `DB_INIT_CONFIRM=I_UNDERSTAND_DB_INIT_APPLIES_SCHEMA_AND_SEED`
- `scripts/deploy_vps_terms_fix.py` ahora falla por defecto salvo que se definan:
